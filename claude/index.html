  • Thinking mode и настройка режимов размышлений
  • Файл CLAUDE.md — главная системная инструкция
  • Уровни CLAUDE.md (локальный / глобальный)
  • Создание реального сайта с подключёнными внешними модулями
  • Скиллы, настройка папки .claude, системные инструкции, роли, агенты
  • Slash-команды, создание собственных скиллов, хуки, MCP-серверы, CLI-инструменты

5. Терминал vs расширение VS Code

Преимущества терминала

  • Контекст: видно процент загрузки контекста в реальном времени
  • Токены: точное количество токенов в контексте
  • Лимиты: отображение оставшихся лимитов (5-часовые, 7-дневные)
  • Кастомизация: custom status line с лимитами
  • Slash-команды: полный набор команд (больше, чем в расширении)
  • Субагенты: переключение между диалогами субагентов
  • Параллельные сессии: можно запустить несколько Claude Code одновременно
  • Bash-режим: команда ! для выполнения терминальных команд прямо из Claude Code
  • Agent Teams: визуальное переключение между агентами

Преимущества расширения VS Code

  • Более привычный, user-friendly интерфейс
  • Удобная интеграция с файловым менеджером
  • Встроенная история диалогов (локальные + веб)
  • Переключение моделей, effort через UI

Ограничения расширения

  • Нет отображения контекста (только иконка при >50%)
  • Нет отображения лимитов
  • Некоторые slash-команды доступны только в терминале
  • Нельзя переключаться между диалогами субагентов

6. Токены, контекст и стоимость

Что такое токены

  • 1 токен ≈ 0.75 слова (примерно 2 токена = 1 слово)
  • Контекст загружается при каждом сообщении

Что попадает в контекст

  • Файл CLAUDE.md (системная инструкция) — при каждом сообщении
  • Все предыдущие сообщения диалога
  • Прочитанные файлы
  • Описания установленных MCP-серверов
  • Краткие описания скиллов (YAML: name + description)

Золотое правило контекста

Одна задача — один диалог. Не перегружать контекст. Желательно не более 50%.

Аналогия: контекст как рабочий день — утром голова свежая (контекст пустой), к вечеру голова забита (контекст переполнен), качество мышления падает.

Последствия переполненного контекста

  • Модель тупеет
  • Лимиты расходуются быстрее
  • Ответы становятся менее точными
  • Стоимость растёт

Управление контекстом — ключевые команды

КомандаДействие
/contextПоказать содержимое контекста с процентным разбиением
/compactСжать диалог в краткое summary
/new или /clearНачать новый диалог с нуля

7. Модели: Opus, Sonnet, Haiku

Переключение модели

  • Терминал: /model
  • Расширение: Switch Model в меню

Сравнение моделей

МодельНазначениеКонтекстСтоимость
Opus 4.6Архитектурные решения, глубокий research, сложные задачи1M токеновВысокая
SonnetИсполнительные задачи, написание кода по готовой спецификацииСтандартныйСредняя
HaikuБыстрые простые задачи, массовая обработка (API)СтандартныйНизкая

Рекомендации по выбору

  • Opus — когда нужно глубокое погружение, высокое качество, изобретательские задачи
  • Sonnet — когда архитектура уже определена, нужно реализовать задачу по описанию
  • Haiku — когда нужна скорость и дешевизна (массовый парсинг, обработка сотен записей)

Лайфхак: Opus Plan

/model opus-plan

Модель автоматически выбирает Opus или Sonnet в зависимости от сложности задачи. Экономит токены, но для критически сложных задач лучше вручную включить Opus.

Thinking Mode и Effort

ПараметрЗначенияВлияние
Thinking ModeOn / OffВключает/выключает режим размышлений
EffortLow / High / Max / AutoГлубина размышлений и расход токенов

Рекомендация: для Opus на сложных задачах — Max effort. Для исполнительных задач — можно снижать.

8. Режимы работы и Bypass Permissions

Четыре режима

РежимОписание
Ask Before EditsПодтверждение каждого действия вручную
Edit AutomaticallyФайлы редактируются автоматически, но инструменты (MCP, терминал) требуют подтверждения
Plan ModeТолько планирование, без изменений файлов
Bypass PermissionsПолная автономия, никаких подтверждений

Включение Bypass Permissions

Способ 1: через settings.json (глобальный или локальный)

{
  "default_mode": "bypass_permissions"
}

Способ 2: флаг при запуске

claude --dangerously-skip-permissions

Безопасность при Bypass Permissions

Обязательно настроить deny permissions в settings.json:

{
  "permissions": {
    "allow": ["Read", "Write", "Bash(...)"],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(export *)",
      "Bash(curl *)",
      "Read(.env)"
    ]
  }
}

Критические запреты:

  • rm -rf * — удаление всего диска
  • export — экспорт секретных ключей
  • curl — отправка данных на внешние серверы
  • Чтение .env — файл с API-ключами и паролями

9. Файл CLAUDE.md — мозг проекта

Назначение

CLAUDE.md — главная системная инструкция проекта. Читается агентом при каждом запуске, перед любым действием.

Что включает

  • Описание проекта и его назначение
  • Архитектура проекта
  • Какие файлы где находятся
  • Технологический стек
  • Дизайн-система (цвета, шрифты)
  • Секции/структура (для сайтов)
  • Соглашения по коду
  • Список агентов и порядок их запуска (для агентных систем)
  • Pipeline взаимодействия агентов

Зачем нужен

При открытии нового диалога агент не знает контекст проекта. CLAUDE.md позволяет ему понять всё из одного файла, не читая каждый файл проекта отдельно.

Уровни CLAUDE.md

  • Локальный — в корне проекта или в .claude/ проекта → действует только для этого проекта
  • Глобальный — в глобальной директории .claude/ → действует для всех проектов

10. Практика: создание сайта с нуля

Порядок работы

  1. Создать пустую папку, открыть в VS Code
  2. Создать файл instructions.md — описание ТЗ (даже в свободной форме)
  3. Загрузить файл в Claude Code через @instructions.md
  4. Попросить агента задать уточняющие вопросы
  5. Ответить на вопросы
  6. Агент создаёт CLAUDE.md (системная инструкция проекта)
  7. После утверждения — агент приступает к реализации

Markdown-файлы — стандарт работы с AI-агентами

  • Формат .md — золотой стандарт коммуникации с агентами
  • Удобно и для человека, и для модели
  • Заголовки, форматирование, структура

Профессиональный подход (для сложных сайтов)

  • Подключение библиотек UI-компонентов
  • MCP-серверы
  • Скиллы для дизайна
  • Техническая документация и спецификация
  • Готовые визуальные элементы

Правило экономии контекста в разработке

  • Написание CLAUDE.md — одна задача (один диалог)
  • Реализация сайта — другая задача (новый диалог)
  • В больших проектах всегда разделять

11. Папка .claude и её структура

Типы папки .claude

ТипРасположениеОбласть действия
ЛокальнаяВ корне проекта .claude/Только текущий проект
ГлобальнаяСкрытая папка на компьютере ~/.claude/Все проекты

Структура папки .claude

.claude/
├── settings.json     # Настройки (permissions, mode, hooks)
├── agents/           # Субагенты (MD-файлы с инструкциями)
├── skills/           # Скиллы (папки с SKILL.md + скрипты)
├── rules/            # Отдельные модули правил
├── worktrees/        # Ветки кода (Git worktrees)
└── CLAUDE.md         # (опционально) системная инструкция

Глобальная директория содержит

  • settings.json — главный файл настроек для всех проектов
  • skills/ — скиллы, доступные во всех проектах
  • sessions/ — диалоги и история
  • plugins/ — плагины Claude Code

settings.json — ключевые настройки

{
  "default_mode": "bypass_permissions",
  "experimental": {
    "agent_teams": true
  },
  "permissions": {
    "allow": [...],
    "deny": [...]
  },
  "hooks": {...},
  "effort_level": "max",
  "model": "opus",
  "theme": "dark",
  "auto_update": "latest",
  "notifications": "terminal_bell",
  "auto_compact": true,
  "verbose_output": true,
  "terminal_progress_bar": true,
  "rewind_checkpoints": true
}

12. Скиллы (Skills)

Что такое скилл

Скилл = инструкция (SKILL.md) + сопутствующие материалы (скрипты, шаблоны, референсы).

В отличие от простого MD-файла, скилл может включать:

  • Python/JS/Shell-скрипты
  • Шаблоны (templates)
  • Референсы (примеры, дизайн-пакеты, Tone of Voice)
  • Assets (активы, ресурсы)

Структура скилла (стандарт Anthropic)

skill-name/
├── SKILL.md          # Главная инструкция скилла
├── scripts/          # Python, JS, Shell-скрипты
├── references/       # Референсные материалы
├── templates/        # Шаблоны
└── assets/           # Активы, ресурсы

YAML-форматирование в SKILL.md

В начале каждого SKILL.md есть скрытый YAML-блок:

---
name: skill-name
description: Краткое описание того, что делает скилл
---

Зачем: агент при каждом запросе читает только name + description всех установленных скиллов (не весь файл). Это экономит контекст. Полный SKILL.md загружается только при вызове конкретного скилла.

Как агент выбирает скилл

  1. Пользователь пишет задачу
  2. Агент читает name + description всех установленных скиллов
  3. Определяет, какой скилл подходит
  4. Загружает полный SKILL.md только нужного скилла
  5. Выполняет задачу по инструкции

Важно: скилл может быть вызван автоматически (без явного указания) или через slash-команду /skill-name.

Установка скиллов

Маркетплейсы:

  • Skills.ai — быстрая установка в 2 клика, сортировка по популярности
  • Smithery — категории (research, coding, planning, design, analytics), указание провайдера

Команда установки:

# Из терминала (не из Claude Code)
claude skills install <repository-url> [skill-name]

При установке выбрать:

  • Платформу (Claude Code, Antigravity, Cursor и др.)
  • Область: Project (локально) или Global (для всех проектов)

Безопасность скиллов

Внимание: проверяйте содержимое скиллов перед установкой! Вредоносные скиллы могут содержать инструкции для отправки API-ключей и паролей злоумышленникам.

Безопасные источники: Anthropic, Vercel, OpenAI, Microsoft, GitHub (официальные).

Примеры скиллов

  • PDF — создание PDF-документов
  • PPTX — создание PowerPoint-презентаций
  • Excel — создание таблиц
  • Agent Browser — автоматизация браузера (скриншоты, парсинг, клики)
  • Skill Creator — создание новых скиллов (от Anthropic)
  • Frontend Design — дизайн интерфейсов

13. Субагенты

Проблема без субагентов

  • Одна задача — один диалог → при сложных задачах контекст переполняется
  • Параллельные диалоги нужно создавать вручную
  • Копирование результатов между диалогами вручную
  • Качество падает при смешении разных задач в одном контексте

Что такое субагент

Субагент — изолированный диалог, который Claude Code запускает автоматически, без участия пользователя. Каждый субагент:

  • Имеет чистый контекст (не загружен историей основного диалога)
  • Загружает только свою инструкцию
  • Выполняет конкретную задачу
  • Возвращает результат главному агенту

Размещение субагентов

.claude/agents/
├── analyst.md
├── coordinator.md
├── reporter.md
├── scraper.md
└── youtube-analyzer.md

YAML-форматирование субагента

---
name: analyst
description: Анализирует собранные данные
tools:
  - Read
  - Write
  - Bash
model: opus
color: green
---

Параметры:

  • name — название агента
  • description — описание роли
  • tools — разрешённые инструменты
  • model — модель (можно Sonnet для простых задач, Opus для аналитики)
  • color — цвет при отображении в терминале

Оркестрация субагентов

Управление описывается в CLAUDE.md (или отдельном pipeline.md):

Pipeline: Координатор → Скрейпер → Аналитик → Репортёр → Координатор

Применение в разработке

Большие проекты разбиваются на чанки (chunks):

  • Каждый чанк — отдельный субагент
  • Внутри чанка — задачи могут разбиваться на ещё более мелкие субагенты
  • Последовательное выполнение: Чанк 1 → новый диалог → Чанк 2 → ...

Переключение между субагентами (только терминал)

В терминале можно стрелками переключаться между:

  • main — основной агент
  • reporter — субагент-репортёр
  • coordinator — субагент-координатор
  • и т.д.

В расширении VS Code это недоступно — видно только запуск и финальный результат.

14. MCP (Model Context Protocol)

Что такое MCP

MCP — протокол от Anthropic, объединяющий все API-запросы к внешним сервисам в единую схему. Аналогия: USB-хаб, через который подключаются внешние сервисы к AI-агенту.

Что даёт MCP

  • Список инструментов сервиса с описанием
  • Структура данных для каждого инструмента
  • Агенту не нужно ничего объяснять — он сам знает, как вызывать инструменты

Альтернатива без MCP

Без MCP нужно вручную описывать каждый endpoint:

  • Структуру запроса
  • Структуру ответа
  • URL, методы, параметры
  • Документацию каждого действия

Установка MCP

Способ 1 — через агента:
Скопировать промт с маркетплейса → вставить в Claude Code → агент сам установит.

Способ 2 — через терминал:

claude mcp install <command>

Маркетплейсы: Smithery, GitHub

Управление MCP

/mcp          # Показать установленные MCP

Проблема MCP: нагрузка на контекст

Каждый MCP загружает описания всех своих инструментов в контекст при каждом сообщении.

Пример: Google Sheets MCP = 38 инструментов ≈ 2000 токенов.
10 MCP-серверов = ~20000 токенов в контексте постоянно.

Лайфхак: конвертация MCP в скилл

Можно попросить Claude Code создать скилл из существующего MCP:

"У нас есть MCP Google Sheets. Проанализируй его документацию и создай скилл, который будет вызываться только по необходимости, используя Skill Creator."

Скилл загружается в контекст только при вызове, в отличие от MCP.

Примеры MCP

  • Google Sheets — управление таблицами (38 инструментов)
  • Chrome DevTools — автоматизация браузера (навигация, клики, скриншоты)
  • Google Calendar — календари
  • YouTube — парсинг видео
  • Notion — управление документами
  • RSS — чтение новостей

15. Agent Teams — агентные команды

Отличие от субагентов

ПараметрСубагентыAgent Teams
КонтекстИзолированныйОбщий (обмен информацией)
КоммуникацияТолько через главного агентаНапрямую между агентами
Расход токеновУмеренныйВысокий
НазначениеПростые параллельные задачиСложные задачи, требующие координации

Включение Agent Teams

В settings.json:

{
  "experimental": {
    "agent_teams": true
  }
}

В CLAUDE.md указать режим работы Agent Teams.

Как работает

  1. Агенты запускаются в параллельных сессиях
  2. Могут обмениваться данными между собой
  3. Могут запрашивать дополнительную информацию у других агентов
  4. Координатор управляет процессом

Визуализация с Tmux

tmux                    # Запуск Tmux
claude                  # Запуск Claude Code внутри Tmux

Tmux позволяет Claude Code автоматически создавать новые окна для каждого агента. Агенты отображаются в отдельных панелях с цветовой кодировкой.

Когда использовать

  • Сложные аналитические задачи с несколькими источниками
  • Задачи, где агентам нужно перепроверять друг друга
  • Когда качество результата важнее стоимости

Предостережение

Agent Teams расходуют значительно больше токенов, чем обычные субагенты. Используйте только для сложных комплексных задач.

16. Агентная система на практике

Архитектура демо-системы

CLAUDE.md (оркестратор)
│
├── agents/
│   ├── router.md          # Координатор, маршрутизатор запросов
│   ├── deep-research.md   # Глубокий research по теме
│   ├── parser.md          # Парсинг веб-сайтов
│   ├── news-digest.md     # Поиск и дайджест новостей
│   ├── doc-analyzer.md    # Анализ документов (Word, PDF, CSV, Excel)
│   ├── report-generator.md # Генерация отчётов (PDF, Sheets, Excel)
│   ├── meeting-notes.md   # Выжимка из записей встреч
│   └── youtube-analyzer.md # Транскрипция и анализ YouTube
│
├── agent-runtime/
│   ├── outputs/           # Готовые отчёты (PDF, HTML, MD)
│   └── shared/            # Данные для обмена между агентами
│
└── skills/                # Установленные скиллы

Маршрутизация запросов (Router)

Ключевые слова в запросеАгент
"найди", "сравни", "исследуй"Deep Research
"собери с сайта", "скачай"Parser
"новости"News Digest
"проанализируй документ" + файлDoc Analyzer
"сделай отчёт"Report Generator
"заметки к встрече"Meeting Notes
YouTube-ссылкаYouTube Analyzer

Цепочки агентов

  • Парсинг + отчёт: Parser → Report Generator
  • Новости + PDF: News Digest → Report Generator
  • YouTube + выжимка: YouTube Analyzer → Report Generator
  • Полный analysis: Parser + News Digest + Deep Research → Report Generator

Принципы написания инструкций агентов

  • Простота: писать настолько просто, насколько возможно
  • Конкретика: описать всё, что нужно, без лишнего
  • Баланс: не больше и не меньше, чем нужно для задачи
  • Chain of Thought: пошаговый алгоритм действий
  • Не усложнять архитектуру: чем сложнее — тем больше точек отказа

17. Деплой и подключение Telegram

Подключение Telegram-бота

Шаг 1: Создать бота в @BotFather

/start → /newbot → имя бота → @username_bot

Получить Bot Token.

Шаг 2: Установить плагин Telegram в Claude Code

# Команда с официального сайта Anthropic
claude plugins install telegram

Шаг 3: Сохранить токен бота

claude telegram set-token "YOUR_BOT_TOKEN"

Шаг 4: Установить Bun (runtime)

  • macOS: curl -fsSL https://bun.sh/install | bash
  • Windows: аналогичная команда

Шаг 5: Запустить в режиме channels

claude --channels

Шаг 6: Написать боту в Telegram → скопировать команду → вставить в Claude Code.

Результат

Полный доступ к агентной системе из Telegram:

  • Запуск агентов
  • Получение отчётов
  • Все функции Claude Code

Ограничение текущей интеграции

  • Работа только с одним диалогом одновременно
  • Компьютер должен быть включён
  • Для нескольких проектов нужна более продвинутая настройка

18. Remote Control с телефона

Настройка

В терминальном Claude Code:

/remote-control

Появится QR-код → отсканировать с телефона в приложении Claude.

Требования

  • Один и тот же аккаунт Claude на компьютере и телефоне
  • Мобильное приложение Claude (раздел Code)
  • Компьютер с активным Claude Code

Преимущества

  • Полный доступ к агентной системе с телефона
  • Все агенты доступны
  • Реальное время — сообщения дублируются

Идеальный сценарий

Claude Code развёрнут на выделенном VPS-сервере (24/7) → доступ с телефона или любого устройства через:

  • Remote Control (мобильное приложение Claude)
  • Telegram-бот
  • Веб-интерфейс (при подключении GitHub-репозитория)

Ключевые принципы работы с Claude Code

  1. Одна задача — один диалог. Не смешивать разные задачи в одном контексте.
  2. Контролировать контекст. Регулярно проверять /context, не превышать 50%.
  3. CLAUDE.md — основа проекта. Всё описание, архитектура, правила — в одном файле.
  4. Скиллы > MCP для редких задач. MCP загружает контекст постоянно, скиллы — по вызову.
  5. Субагенты для больших задач. Разбивать сложные задачи на изолированные субагенты.
  6. Agent Teams для комплексных задач. Когда агентам нужно обмениваться информацией.
  7. Безопасность: настраивать deny permissions, проверять скиллы перед установкой, не хранить секреты в доступных файлах.
  8. Правильная модель для задачи: Opus для сложного, Sonnet для исполнения, Haiku для массовой обработки.
  9. Markdown — стандарт коммуникации с AI-агентами.
  10. Простота инструкций: конкретно, но без лишнего. Чем проще — тем надёжнее.
